# 积木：Trade-off 优劣权衡

## 适用

RESOURCE_COMPARE / METHOD_COMPARE 的提炼型积木。当多个对象（数据集 / 方法 / 工具 / baseline）共存且没有严格"全面碾压"关系时，整理优劣权衡 + 选型/联用建议。

⚠️ **必须**与某个陈列型积木（matrix / resource-table / ...）配合使用——权衡是从陈列中提炼的，不是凭空写的。

## micro-prompt

1. 从陈列型积木中确定 2-N 个被对比的对象（数据集名 / 方法名 / 工具名）
2. 为每个对象写：
   - 2-4 条 **优势**
   - 2-4 条 **劣势**
   - 每条必须有具体依据：数值、场景、或 [[论文]] 引用
3. 写一段 **选型/联用建议**：
   - 二者关系：替代 / 互补 / 部分重叠
   - 何种场景该用谁
   - 是否应联用（如有，给依据论文）

## 约束

- **平衡**：每个对象的优势/劣势数量应大致对称（差距不超过 1）。一边写 4 条另一边写 0 条 = 隐含偏见，不可接受
- **具体**：每条 ≤ 1.5 行，禁用"效果好" / "有局限" / "性能好"等空话
- **依据**：每条优势/劣势在 report 内部要可反查到陈列积木中某个单元格 / 某条观察
- **不凑数**：宁可 2 条强论据，不要 4 条掺水
- **建议必须答"何时用谁"**：禁用"看情况" / "根据需求选择"等回避表达

## micro-template

```markdown
### 优劣权衡

**{{对象 A}}的优势**
1. ...
2. ...

**{{对象 A}}的劣势**
1. ...
2. ...

**{{对象 B}}的优势**
1. ...
2. ...

**{{对象 B}}的劣势**
1. ...
2. ...

{{若有更多对象继续重复，N ≤ 5}}

**选型 / 联用建议**
{{2-4 句：明确说"X 适合做 Y 时用"、"二者互补/替代"、"是否应同时报告"。
若有论文已示范联用方式，引用 [[作者_年份_关键词]] 作依据}}
```

## 反例

❌ 优势写"规模大" → 应写"5.4 万条，比 Jingfang-SD 多 1 倍"
❌ 劣势栏空着 → 任何方案都有 trade-off，劣势栏空 = 隐含偏见
❌ 联用建议写"根据具体需求选择" → 必须明确"做长尾评估时用 X，做流派评估时用 Y"
❌ 一边 4 条优势，另一边 1 条优势 → 失衡，要么补足要么删减
❌ 优势/劣势条目无 [[论文]] 或具体数值依据 → 等于无法被读者审计

## 设计起源

v0.2 引入。从 `wiki/reports/证候数据集对比_2026-04-14.md` 的"自由权衡段"中浮现的稳定模式：4 个并列 block（A 优 / A 劣 / B 优 / B 劣）+ 联用建议段。该模式在第一次替代使用中即表现出明确的结构性，符合"反馈累积 ≥ 3 次再沉淀"原则的早期触发——因为这里的结构信号足够强（不是凑出来的）。
