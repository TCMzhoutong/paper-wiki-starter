# 积木：因果归因 / 驱动因素分析

## 适用

EVOLUTION / ATTRIBUTION 的提炼型积木。当问题需要解释"为什么 X 演变到 Y"或"是什么推动了 Z"时，逐节点拆解驱动力。

⚠️ **必须**与某个陈列型积木（matrix / timeline / ...）配合使用——归因是从陈列出的事实序列中提炼的。

## micro-prompt

1. 从陈列积木中识别 2-N 个**转折/升级节点**（如：方法 1 → 方法 2 / 范式 A → 范式 B）
2. 对每个节点写 Push/Pull/结果三段：
   - **Push**：前代方案的具体失败（可量化最佳）
   - **Pull**：新方案的外部可能性来源（技术成熟、邻域成功、工具/数据可得）
   - **实际结果**：升级是否真的解决了 Push 中的问题？给数据或定性结论
3. 可选追加（按需）：
   - **策略分裂**：同一升级方向上是否存在不同流派？（如"反思控噪 vs 个性化适配"）
   - **关键转折**：本次升级最不可逆的认知转变是什么？（如"从匹配文本转向匹配结构"）

## 约束

- **节点数控制**：2-5 个节点最佳，超过 5 个变流水账
- **Push 必须具体**：禁用"性能不够"、"效果有限"，要给数值或场景
- **Pull 必须外向**：是什么**外部条件**让升级成为可能（不是"我们觉得需要"）
- **结果必须诚实**：如果升级**没有**解决 Push 的问题，如实写"失败"——这是 attribution 最有价值的部分
- **不要假设单调进步**：明确写出哪些升级是"成功"，哪些是"暴露新问题"

## micro-template

```markdown
### 驱动因素分析

{{1 句开场：本次演进的总体特征，如"每次升级都是被前一代具体失败推动的，不是渐进改良"}}

**{{节点 1：A → B}}的驱动**
- *Push*：{{前代具体失败，可带数值}}
- *Pull*：{{外部新可能性}}
- *实际结果*：{{是否解决 Push 中的问题；给数据或结论}}
- *策略分裂*（可选）：{{同方向上的不同流派/思路}}
- *关键转折*（可选）：{{本次不可逆的认知转变}}

**{{节点 2：B → C}}的驱动**
- *Push*：...
- *Pull*：...
- *实际结果*：...

{{重复 N 次，N ≤ 5}}
```

## 反例

❌ Push 写"传统方法效果不好" → 应写"GPT-4 加 RAG 后准确率 55.83% → 49.83%（[[Liu_2026]] 报告）"
❌ Pull 写"我们想到了用 KG" → 应写"通用 Graph RAG 工作（LlamaIndex 等）成熟"
❌ 所有节点都标"实际结果：成功" → 几乎不可能；至少有一个节点的"成功"应是部分的或暴露了新问题
❌ 节点 ≥ 6 个 → 流水账，应合并相近的转折

## 设计起源

v0.3 引入。从 `wiki/reports/知识增强技术路径_2026-04-14.md` 的"驱动因素分析"段中浮现。在 4 个升级节点上**一次性**写出稳定的 Push/Pull/结果三段结构，符合"结构信号足够强"的早期沉淀判据（同 tradeoff）。"策略分裂"作为可选段保留，因为它在 3→4 节点上首次出现并起关键作用，但不强制每个节点都有。
