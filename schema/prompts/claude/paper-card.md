# 生成论文精读卡片 — Claude 格式化模板

## 输入

两类数据源：

1. **元数据**（来自 endnote-cli）：title、authors、year、venue、doi、reference_type、type_of_work。通过 `endnote-cli search quick "<标题关键词>"` 或 `endnote-cli item get <id>` 获取
2. **内容分析**（来自 NotebookLM）：4 份笔记（结构化摘要、方法细节、实验与结果、关键概念）。其中"关键概念"笔记每条包含：中文规范名、英文全称、常见缩写、定义、角色、关联

## 输出

- `wiki/papers/作者_年份_关键词.md`（论文精读卡片）
- `wiki/concepts/*.md` 下若干新建 stub 或 alias 回写（见阶段 C）

## 核心原则

精读卡片是**详尽的精读笔记，不是摘要**。必须：

- **完整保留** NotebookLM 分析的所有细节，不要压缩或省略
- 方法部分要详细拆解每个模块，不能一句话概括
- 实验结果要列出所有具体数值和对比
- 概念关系要完整呈现，不能只列名称
- frontmatter 的 `title`、`authors`、`year`、`venue`、`doi` 使用 endnote-cli 元数据
- frontmatter 的 `tldr` 由 LLM 根据内容分析生成（一句话概括核心贡献）
- frontmatter 的 `tags` 分两部分：
  - **论文类型**（从 endnote-cli 映射）：按下表取第一个命中的

    | 条件 | tag |
    |---|---|
    | `type_of_work` 含 "Review" | 综述 |
    | `venue` 含 "arXiv" 或 `reference_type`=47 | 预印本 |
    | `type_of_work` 含 "博士"/"硕士" 或 `reference_type`=31 | 学位论文 |
    | `reference_type`=3 或 4 | 会议论文 |
    | `reference_type`=1 | 书籍 |
    | `reference_type`=2 | 书章节 |
    | `reference_type`=0（默认） | 期刊论文 |

  - **研究性质**（LLM 判断内容追加）：方法论、实证研究等
- 研究领域和方法不放 frontmatter，由正文 `[[wikilink]]` 表达

## 执行流程：三阶段

处理每篇新论文时，必须按 A → B → C 顺序执行。

### 阶段 A — 建立规范名映射（开头一次性）

在生成论文卡片前，加载现有的术语体系。根据概念数量选择方式：

**≤200 概念**：直接加载
1. `Glob wiki/concepts/*.md` — 获取所有规范文件名
2. `Grep "^aliases:" wiki/concepts/` — 获取所有别名行
3. 在上下文里建立 `{ 别名 → 规范名 }` 映射表

**>200 概念**：使用索引
1. `python scripts/concept-audit.py index` 构建索引
2. 读取 `scripts/concept-index.json`（`{ 变体名小写 → 规范名 }` 查找表）
3. 对 NotebookLM 返回的每个概念名，在索引中查找匹配

若 `wiki/concepts/` 为空（首篇论文），跳过此阶段。

### 阶段 B — 生成卡片时的规范化（策略 A 保守）

对 NotebookLM "关键概念"笔记里的每个概念，按优先级决定如何写 `[[X]]`：

1. **精确命中规范文件名** → 直接用 `[[规范名]]`
2. **命中某个 alias** → 用 `[[alias 对应的规范名]]`
3. **中英文对译或标准缩写的明显等价**（如 `LLM` ↔ `大型语言模型`、`GPT` ↔ `Generative Pre-trained Transformer`）→ 用 `[[已有规范名]]`，并标记此变体需要 **alias 回写**
4. **其余所有情况** → 作为新概念处理，用 `[[中文规范名]]`，标记需要 **stub 创建**

**策略 A 保守原则**：只合并明显的同义变体（中英文对译、标准缩写）。以下情况**不合并**，视为不同概念：

- `BERT` vs `BERT-base` vs `SciBERT`（家族 vs 具体变体）
- `LLM` vs `Llama 3`（类别 vs 实例）
- `注意力机制` vs `自注意力机制`（泛称 vs 特化）

宁可创建两个相关概念页用 `related` 字段链接，也不要错合并。

### 阶段 C — 写入卡片后的扫尾

论文卡片写入 `wiki/papers/` 后，按阶段 B 标记的列表处理：

1. **对"alias 回写"的概念**：`Edit` 对应的 `wiki/concepts/规范名.md`，将新变体追加到 `aliases` 字段（只追加不覆盖）
2. **对"stub 创建"的概念**：按 `concept-update.md` 的 Stub 模板新建 `wiki/concepts/新概念.md`，填入定义、aliases（英文+缩写）、相关概念

不需要编辑概念页的"相关论文"——论文卡片中的 `[[概念名]]` 会被 Obsidian 自动追踪为反向链接。

两类操作可以并行。所有操作完成后，`wiki/concepts/` 应与本次论文的所有 `[[X]]` 引用一一对齐（要么是已有规范名/alias，要么是新建的 stub）。

## 模板

frontmatter 见 `schema/prompts/wikilink-format.md` 的"论文卡片"章节。正文结构：

```markdown
# {{论文简称}}

## 核心贡献

{{1-3 句话概括本文最重要的贡献，关键术语用 [[双链]]}}

## 研究背景

{{完整的研究动机、问题定义、现有方法的不足，保留 NotebookLM 分析的所有细节}}

## 方法

{{方法整体概述}}

### 整体框架

{{完整描述 pipeline 的每个阶段，保留所有模块细节}}

### 模块详解

{{逐个拆解每个核心模块：}}
{{- 模块名称}}
{{- 输入是什么}}
{{- 处理过程（算法/模型/参数设置）}}
{{- 输出是什么}}
{{- 与其他模块的关系}}

### 关键创新点

{{逐条列出创新点及其具体说明}}

### 与 baseline 的区别

{{详细对比本文方法与对比方法的差异}}

## 实验设计

### 数据集

{{数据集来源、规模、划分方式、标注方法等完整信息}}

### 评估指标

{{列出所有评估指标及其定义}}

### 实验设置

{{模型参数、训练配置、硬件环境等}}

## 实验结果

### 主实验

{{用表格呈现完整的性能对比，包含所有方法和所有指标的具体数值}}

| 方法 | 指标1 | 指标2 | ... |
|---|---|---|---|
| ... | ... | ... | ... |

### 消融/分析实验

{{逐条列出消融实验或因素分析的结论，保留具体数值}}

### 关键发现

{{逐条列出重要发现，每条包含支撑数据}}

## 局限性

{{逐条列出，每条详细说明}}

## 未来方向

{{逐条列出作者提出的未来工作方向}}

## 关键概念

{{列出所有关键概念，每个用 [[双链]]，附完整定义和在本文中的角色}}

- [[概念1]] — 完整定义 | 角色: xxx | 关联: [[概念2]], [[概念3]]
- [[概念2]] — 完整定义 | 角色: xxx | 关联: [[概念1]]
- ...

## 概念关系图

{{用文字描述概念之间的关系网络，每条关系单独一行：}}
- [[概念A]] →（改进了）→ [[概念B]]
- [[概念C]] →（用于评估）→ [[概念D]]
- ...

```

## 执行规则

1. **不要压缩** — NotebookLM 返回的细节必须完整保留，只做结构化整理
2. 参考 `schema/prompts/wikilink-format.md` 确保 frontmatter、双链、命名合规
3. 实验结果表格必须包含**所有方法**和**所有指标**的具体数值
4. 方法描述要逐模块拆解，不能合并为一段话
5. 概念部分要包含完整定义，不能只写名称
